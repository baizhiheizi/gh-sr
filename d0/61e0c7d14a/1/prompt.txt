You can access the self-hosted runner by `ssh x1` to review.

What I found so far:

What I Found (in chronological investigation order)
1. /home/runner/ missing
The gh-aw lock file hardcodes /home/runner/ (matching GitHub-hosted runners where the user is literally runner). On x1, the runner user is an-lee, so /home/runner/ didn't exist. Fixed by user.

2. host.docker.internal not resolvable
The MCP gateway runs with --network host and connects to the safeoutputs server at http://host.docker.internal:PORT. Linux doesn't have this by default. Fixed by user.

3. gh not authenticated in the AWF agent container
The agent runs inside an AWF Docker container (chrooted on the host filesystem), not directly on the host. The gh token stored in GNOME Keyring on the host is not accessible inside the container. This is why gh auth status says "not logged in" inside the agent.

Confirmed: When I tested from x1:

gh auth token works on the host (token: gho_...)
Inside the AWF agent container, GH_TOKEN env var is stripped (empty string) and the container has no access to the host's keyring
GH_TOKEN passed via -e GH_TOKEN also gets cleared inside the AWF agent container
4. MCP Gateway Startup Actually Succeeds
The "Start MCP Gateway" step logs show:

Gateway container starts successfully
Health check passes: {"status":"healthy","servers":{"github":{"status":"running"},"safeoutputs":{"status":"running"}}}
Configuration converts and writes to /tmp/gh-aw/mcp-config/mcp-servers.json
5. But Agent Still Reports MCP Failure
Despite the gateway showing "running", the agent reports ERR_API: MCP server(s) failed to launch.

6. host.docker.internal unreachable from AWF container
Even with the gateway running and the host correctly resolving host.docker.internal, the AWF agent container cannot reach host.docker.internal:80. I confirmed this with curl -v http://host.docker.internal:80/health from inside the AWF container — Connection refused.

This means: when the AWF agent container starts and tries to connect to the MCP gateway at http://host.docker.internal:80, it gets refused.
